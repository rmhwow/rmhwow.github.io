SQL Injection

What is SQL Injection?

Nobody likes a glitch, but worse is a hacker. SQL Injection is a code technique where malicious SQL statements are inserted into entry fields. One popular example of this is dump all the database contents to the attacker as illustrated by this fun comic. It's considered one of the top attacks on web application. The five main sub-classes of SQL Injection Include:

  1. Classic SQLI
  2. Blind or Interference SQL Injections
  3. Database management-system -specific SQLI
  4. Compounded SQLI
    • SQL injection + insufficient authentication
    • SQL injection + DDoS attacks
    • SQL injection + DNS hijacking
    • SQL injection + XXS

Houston, We Have a Problem

So What's the big deal? Well, when a hacker writes malicious code it gives the hacker direct communication to your databases. This can allow the hacker to retrieve every bit of information stored in that database and wipe it out. This can be done through email, login page, manipulating URLS etc. Some of the most significant attacks include Storm Worm. An injection that attacked thousands of computers in 2007. In January of 2007 Storm Worm accounted for 8% of all malware infections GLOBALLY.

How it is done?

As with everything, there is hope! In order to understand how we can prevent it, we need to look at the potential areas for hacking. This can be anything where the web application would store user information. A great example of this would be a login page. For code purpposes the most important part of a login in page is the "username" and password string sql= "SELECT * FROM User_login WHERE username = '"+username+'' and password = '"+password+"' This means that User_login is the datatable where username and password are stored. Now if someone put in 1=1;-- as the username and "" They would be able to access someone's account. Reason being that when the code sees username= 1=1 that is automatically going to come back as true and the password is commented out.

How Can it Be Prevented?

While programmers continue working on this problem there are a few helpful hints to prevent attacks.

  1. Assume all end user input is hostile
  2. Don't have too detailed error messaging. It could make it easier for a hacker
  3. Minimize the input data length
  4. Don't depend on client side validation. It can definitely be bypassed
